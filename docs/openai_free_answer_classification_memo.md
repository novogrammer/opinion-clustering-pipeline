# OpenAI APIを使った自由回答分類の整理メモ

## 前提

自由回答の分類ラベルが事前に決まっていない場合は、いきなり全件を分類するのではなく、まず **意味が近い回答を集める**。

基本の流れは以下。

```txt
元データ
↓
Codexで分析用データ形式に変形
↓
分類対象を機械判定
↓
自由回答
↓
OpenAI Embeddings APIでベクトル化
↓
BERTopicなどでクラスタリング
↓
各クラスタの代表回答を抽出
↓
OpenAI APIでクラスタ名・定義を作成
↓
人間がカテゴリを整理
↓
カテゴリ定義をベクトル化
↓
回答 embedding とカテゴリ embedding の近さで全件分類
↓
集計・レポート化
```

ここでの機械判定は、空欄、定型無回答、記号のみのように文字列ルールで落とせるものだけを先に切り分ける想定とする。

---

## ベクトル化とは

ベクトル化とは、文章をOpenAIのEmbeddingモデルに渡して、数値配列に変換すること。

例：

```txt
料金が高いので、もう少し安くしてほしい
```

↓

```txt
[0.012, -0.045, 0.331, ...]
```

この数値配列が、その文章の意味を表すベクトル。

次元数は自分で設計するものではない。  
使うEmbeddingモデルが決める。

---

## 使うモデル

まずは以下でよい。

```txt
text-embedding-3-small
```

理由：

- 安い
- 速い
- 1万〜3万件規模の自由回答分類にはまず十分
- 分類候補を見つける用途なら扱いやすい

より微妙なニュアンス差を拾いたい場合は、比較用に以下も検討する。

```txt
text-embedding-3-large
```

ただし、同じ分析内ではモデルを混ぜない。

```txt
OK:
全件 text-embedding-3-small

NG:
質問1だけ small、質問2だけ large
```

---

## 質問文も含めてベクトル化する

自由回答では、同じ回答でも問いによって意味が変わる。

例：

```txt
Q. よかった点は？
A. 安い
```

→ 価格への満足

```txt
Q. 不安な点は？
A. 安い
```

→ 品質への不安かもしれない

そのため、ベクトル化するテキストには、回答だけでなく質問も含める。

```txt
設問ID: Q1
質問: 今回よかった点を教えてください
回答: 安い
```

これを1つのテキストとしてEmbedding APIに渡す。

---

## 件数の考え方

質問1に1万件、質問2に1万件、質問3に1万件ある場合、合計3万件分のベクトルを作る。

```txt
質問1: 1万ベクトル
質問2: 1万ベクトル
質問3: 1万ベクトル

合計: 3万ベクトル
```

ただし、API呼び出しを3万回するわけではない。  
100件ずつ、500件ずつなど、バッチでまとめて処理する。

---

## 質問ごとにクラスタリングする

ベクトル化は3万件分行うが、クラスタリングは質問ごとに分ける。

```txt
質問1の1万件 → クラスタリング
質問2の1万件 → クラスタリング
質問3の1万件 → クラスタリング
```

質問をまたいで全部混ぜると、問いの違いによるニュアンスが混ざる可能性がある。

最初は質問ごとに分類し、必要なら最後に横断カテゴリを作る。

---

## OpenAI Embeddings APIの使い方

Pythonでは以下のように使う。

```python
from openai import OpenAI

client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input=[
        "設問ID: Q1\n質問: よかった点は？\n回答: 安い",
        "設問ID: Q1\n質問: よかった点は？\n回答: 使いやすい",
    ]
)

embeddings = [item.embedding for item in response.data]
```

この `embeddings` がベクトル配列。

---

## ベクトルは保存する

Embedding APIで作ったベクトルは保存しておく。

```txt
responses.csv
embeddings_q1.npy
embeddings_q2.npy
embeddings_q3.npy
```

一度保存しておけば、BERTopicのパラメータを変えて試すたびにOpenAI APIを呼び直す必要がない。

つまり、

```txt
ベクトル化は基本1回
クラスタリングは何度でも試せる
```

という状態にできる。

---

## BERTopicとの関係

OpenAI APIはベクトル化を担当する。  
BERTopicは、そのベクトルを使ってクラスタリングする。

```txt
OpenAI Embeddings API
= 文章をベクトルにする

BERTopic
= ベクトルを使って似た回答をグルーピングする
```

BERTopicには、OpenAIで作ったembeddingを渡せる。

```python
topics, probs = topic_model.fit_transform(docs, embeddings)
```

---

## 高次元でもクラスタリングできる理由

OpenAIのEmbeddingは高次元のベクトルになる。

ただし、そのまま人間が扱うわけではない。  
BERTopicでは通常、以下のように処理される。

```txt
OpenAI Embedding
↓
UMAPで次元削減
↓
HDBSCANでクラスタリング
↓
トピック番号を付与
```

つまり、高次元ベクトルをいったん扱いやすい次元に圧縮してからグルーピングする。

---

## クラスタリング後の名付け

クラスタリング後は、各クラスタから代表回答を取り出す。

代表回答は、中心に近い回答だけでなく、端の回答や少数派も混ぜる。

OpenAI APIに以下を依頼する。

```txt
以下は同じクラスタに入った自由回答です。

回答例:
- 料金が高い
- 月額費用がもう少し安いと助かる
- 価格に対して機能が少ない
- 導入費用がネック

依頼:
1. 分類名を付けてください
2. 分類定義を書いてください
3. 含める回答の基準を書いてください
4. 含めない回答の基準を書いてください
5. 代表回答を3件選んでください
6. 複数テーマが混ざっていれば分割案を出してください
7. 自信度を High / Medium / Low で付けてください
```

クラスタ名はAIに初案を出させる。  
ただし、そのまま最終分類にはしない。

---

## 人間が整理する部分

AIとBERTopicでできるのは、あくまで「分類候補の発見」。

最終的には人間が以下を行う。

```txt
似たカテゴリを統合する
粒度が粗すぎるカテゴリを分割する
業務上使いやすい名前に直す
カテゴリ定義を整える
その他・低信頼度の扱いを決める
```

分類は自然に存在するものではなく、目的に応じて設計するもの。

---

## 確定カテゴリで全件分類する

カテゴリ表が固まったら、カテゴリ側も同じ Embedding モデルでベクトル化する。

この段階では、回答側の既存 embedding とカテゴリ側 embedding の近さを使う。

```txt
入力:
- 質問
- 回答
- 確定カテゴリ一覧

出力:
- 該当カテゴリ
- confidence
- 理由
```

カテゴリ側 1 件を 1 ベクトルとして扱い、最も近いカテゴリを採用する。  
confidence が低い場合だけ `OTHER` や人手確認に回す。

---

## 実務フロー

```txt
1. 質問ごとに回答を分ける
   Q1: 1万件
   Q2: 1万件
   Q3: 1万件

2. 元データCSVをCodexで分析用データ形式に変形する
   必須列をそろえる
   設問マスタを結合する
   複数CSVファイルを必要に応じて統合する

3. 無回答や分類対象外を切り分ける

4. 設問ごとに処理を分ける

5. 各回答について、ベクトル化用テキストを作る
   設問ID
   質問
   回答

6. OpenAI Embeddings APIで3万件分をベクトル化する

7. ベクトルを保存する

8. 質問ごとにBERTopicでクラスタリングする

9. 各クラスタの代表回答を抽出する

10. OpenAI APIでクラスタ名・定義・分割案を作る

11. 人間がカテゴリを整理する

12. 確定カテゴリを同じ Embedding モデルでベクトル化する

13. 回答 embedding とカテゴリ embedding の近さで全件分類する

14. Low confidence や その他カテゴリを人間が確認する

15. 集計・レポート化する
```

---

## 役割分担

```txt
OpenAI Embeddings API
= 文章を意味ベクトルに変換する

BERTopic
= 似た回答をクラスタリングする

OpenAI LLM
= クラスタ名、定義、分割案の草案を作る

Codex
= 元データを分析用データ形式に変形する

人間
= 分類粒度、統合・分割、最終カテゴリを決める
```

---

## 重要な注意点

Embeddingは分類そのものではない。  
意味の近さを扱うための下処理。

BERTopicのクラスタは最終分類ではない。  
似た回答の山を見つけるためのもの。

OpenAI APIによる名付けも最終判断ではない。  
分類表として使える粒度にするには、人間の整理が必要。

最終的には、

```txt
Embeddingで近さを見る
BERTopicで山を作る
OpenAIでクラスタ草案を作る
人間が分類表として整える
カテゴリ embedding と回答 embedding の近さで全件に分類を付ける
```

という流れで考える。
