# 自由回答分類の知見メモ

この文書は、実装仕様の背景説明と運用上の知見をまとめる補足資料とする。

実装仕様そのものは [pipeline_spec.md](./pipeline_spec.md) を参照する。
レビュー基準は [review_criteria.md](./review_criteria.md) を参照する。

## 基本的な考え方

自由回答の分類ラベルが事前に決まっていない場合は、いきなり全件を分類するのではなく、まず意味の近い回答を集めて分類候補を発見する。

大まかな役割分担は以下。

```txt
OpenAI Embeddings API
= 文章を意味ベクトルに変換する

k-means
= 似た回答をクラスタリングする

Codex
= 元データを分析用データ形式に変形する

人間
= 分類粒度、統合・分割、最終カテゴリを決める
```

Embedding は分類そのものではない。意味の近さを扱うための下処理であり、k-means のクラスタも最終分類ではない。

## 採用方針の背景

### `text-embedding-3-small` を初期採用にする理由

- 速い
- 安い
- 1万〜3万件規模の自由回答分類ではまず十分
- 分類候補の発見用途なら扱いやすい

より微妙なニュアンス差を拾いたい場合は `text-embedding-3-large` の比較もありうるが、同じ分析内でモデルを混ぜない方がよい。

### 質問文も含めてベクトル化する理由

同じ回答でも問いが変わると意味が変わる。

```txt
Q. よかった点は？
A. 安い
```

これは価格への満足かもしれない。

```txt
Q. 不安な点は？
A. 安い
```

こちらは品質への不安かもしれない。

そのため、回答だけでなく設問文脈もまとめてベクトル化する方が実務上自然になる。

### 設問ごとにクラスタリングする理由

複数設問をまとめてクラスタリングすると、問いの違いによるニュアンス差が埋もれやすい。

最初は設問単位でクラスタリングし、必要なら最後に横断カテゴリを人が作る方が扱いやすい。

### ベクトルを保存する理由

Embedding を保存しておけば、k-means の `k` を変えたり HDBSCAN と比較したりしても、毎回 API を呼び直さずに済む。

```txt
ベクトル化は基本1回
クラスタリングは何度でも試せる
```

## 技術メモ

### ベクトル化とは

文章を Embedding モデルに渡して、数値配列に変換すること。

```txt
料金が高いので、もう少し安くしてほしい
```

↓

```txt
[0.012, -0.045, 0.331, ...]
```

この配列が文章の意味を表すベクトルになる。次元数はモデル側で決まる。

### k-means との関係

k-means は OpenAI で作った embedding を受け取り、近いベクトルを指定数のクラスタへまとめる。

標準想定は約1万回答、`k=100` で、平均は約100回答 / クラスタになる。回答件数や必要なカテゴリ粒度が大きく異なる場合だけ `--k` を調整する。

標準フローは以下。

```txt
OpenAI Embedding
↓
k-meansでクラスタリング
↓
トピック番号を付与
```

HDBSCAN は比較用オプションとして残し、明示的に選んだ場合だけ UMAP + HDBSCAN の構成を使う。

### OpenAI Embeddings API の最小例

```python
from openai import OpenAI

client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input=[
        "設問ID: Q1\n質問: よかった点は？\n回答: 安い",
        "設問ID: Q1\n質問: よかった点は？\n回答: 使いやすい",
    ],
)

embeddings = [item.embedding for item in response.data]
```

## 実測メモ

以下は、ある設問 `Q1` を `text-embedding-3-small` でベクトル化したときの実測メモ。  
仕様ではなく、処理時間とコスト感の目安として残す。

前提:

- 対象件数: `8901`
- 入力形式: `設問ID + 質問文 + 回答文`
- 出力次元数: `1536`
- 実行日: `2026-07-05`

処理時間:

- `100件`: 約 `5秒`
- `1000件`: 約 `12.2秒`
- `8901件`: 約 `88.5秒`

見方:

- 件数に比例する時間だけではなく、数秒の固定オーバーヘッドがある
- `100件` の結果をそのまま 100 倍するより、`1000件` を挟んで見積もる方が精度が高い
- 今回の実測では、`1万件` 規模でも数分以内に収まる見込みだった

入力量の概算:

- 総入力文字数: 約 `83.3万文字`
- 1件あたり平均: 約 `93.6文字`
- token 概算: 約 `45万-60万 token`

コスト感:

- `Q1` 実行後、Billing 上の増分は `0.01ドル`
- したがって、この規模の設問 1 本あたりの embedding コストは数セント未満と見てよい

注意:

- token 数は API レスポンスから直接取得した値ではなく、文字数ベースの概算
- 料金は将来変わりうるため、実運用前には最新の pricing と Billing で再確認する

過去の HDBSCAN クラスタリング実測:

- 対象件数: `8901`
- 所要時間: 約 `58.7秒`
- topic 数: `210`
- outlier `-1`: `1955件`

見方:

- embedding 作成よりは重いが、1万件弱なら 1 分前後で終わる規模だった
- この値は比較用 HDBSCAN の実測であり、標準の k-means では outlier `-1` は生成されない
- HDBSCAN を使う場合は、`05_curation` で通常 topic だけでなく outlier の量も確認対象になる
- topic 数 `210` はそのまま最終カテゴリ数を意味しない。ここから人が統合して業務用カテゴリに整理する前提

---

## クラスタリング後の人手整理

クラスタリング後は、各クラスタから代表回答を取り出す。

代表回答は `topic_probability` の高い順に抽出し、クラスタ中心に近い典型的な回答を確認する。あわせて `topic_size` を見て、クラスタの大きさを確認する。

---

## 人間が整理する部分

Embedding と k-means でできるのは、あくまで「分類候補の発見」。

最終的には人間が以下を行う。

```txt
似たカテゴリを統合する
粒度が粗すぎるカテゴリを分割する
業務上使いやすい名前に直す
カテゴリ定義を整える
その他・低信頼度の扱いを決める
```

分類は自然に存在するものではなく、目的に応じて設計するもの。

---

## 確定カテゴリで全件分類する考え方

カテゴリ表が固まった後は、人が確定した `topic_id -> category_id` の対応を全回答へ再適用する。

HDBSCAN を比較利用して外れ値 topic が生じた場合は、`OTHER` や人手確認へ回す前提が扱いやすい。

---

## 概念フロー

```txt
元データを分析用形式にそろえる
↓
無回答や対象外を切り分ける
↓
設問文脈つきでベクトル化する
↓
設問ごとにクラスタリングする
↓
人がカテゴリを整理する
↓
`topic_id -> category_id` を全件へ再適用する
↓
`OTHER` を確認して集計する
```

## 重要な注意点

Embeddingは分類そのものではない。  
意味の近さを扱うための下処理。

k-meansのクラスタは最終分類ではない。
似た回答の山を見つけるためのもの。

最終的には、

```txt
k-meansでクラスタを作る
人間が分類表として整える
人が決めた topic と category の対応を全件へ再適用する
```

という流れで考える。
